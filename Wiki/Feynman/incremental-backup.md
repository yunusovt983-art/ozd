# Инкрементальный бэкап + общий refcount — «методом Фейнмана»

## Проблема

[Бэкап в cold_path/S3](dfs-backup.md) копирует данные. Но датасет — **480 ТБ**. Копировать **всё
целиком** каждый раз — безумие (часы/сутки, гигантский трафик). А между бэкапами меняется лишь
небольшая часть. Как бэкапить **только изменения** — и при этом безопасно удалять старые бэкапы?

## Аналогия — «Машина времени», а не полная копия диска каждый день

Time Machine не копирует весь диск ежедневно — копирует **только новые/изменённые файлы**, а
неизменные **переиспользует** из прошлых бэкапов (через ссылки):

1. 📦 **Грузить только новое.** Бэкап-точка заливает в cold-store **только сегменты, появившиеся/
   изменившиеся** с прошлой точки. Сегменты [иммутабельны](immutability.md) и content-addressed →
   «не менялся» = «то же имя/CID» → **дедуп бесплатно** (не грузим повторно).
2. 🔗 **Общие сегменты — по ссылке.** Сегмент, который нужен **нескольким** бэкап-точкам, лежит в
   cold-store **один раз**; точки на него ссылаются.
3. 🔢 **Refcount: кто ещё держит?** У каждого сегмента в cold-store — счётчик «сколько бэкап-точек на
   него ссылаются». Удаляем устаревшую точку → уменьшаем счётчики; сегмент **физически стираем, лишь
   когда его не держит ни одна** точка (как [reader-watermark](reader-watermark-cleaner.md), но на
   уровне бэкапов).
4. 🧱 **Полный бэкап — это просто база.** Первая точка (или периодическая «база») = полный
   [DFS-бэкап](dfs-backup.md); дальше — инкременты поверх неё.

> Чертёж из Flink (incremental checkpoint: грузить только новые SST; `SharedStateRegistry` рефкаунтит
> файлы между чекпойнтами, async-discard при refcount=0).

## Проверка понимания

> **Инкрементальный бэкап** = заливать в cold-store **только новые сегменты** (иммутабельность даёт
> дедуп по CID), общие сегменты держать по **refcount** и стирать лишь когда **ни одна** бэкап-точка
> их не держит.

## Зачем это нам

Фаза 5: бэкап 480 ТБ дельтой, а не полной копией; безопасная очистка старых точек.
