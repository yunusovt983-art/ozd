# MoveTs read-fence при переезде блоков — «методом Фейнмана»

## Проблема

Иногда блоки **переезжают** с диска на диск: добавили новый диск, балансировщик выравнивает
заполнение, сменился HRW-владелец зоны. Во время переезда есть опасный момент: кто-то приходит
читать, а данные **уже наполовину там, наполовину тут**. Можно прочитать **старую** раскладку и
получить «дыру» (блок на старом диске уже удаляют, на новом ещё не зафиксирован).

## Аналогия — переезд офиса с табличкой «работаем с такого-то числа»

Отдел переезжает в новый кабинет. Чтобы не было путаницы, на новой двери вешают табличку:
**«принимаем посетителей с 10:00»**.

1. 🚚 **Сначала перевозим, потом открываем.** Пока коробки едут, новый кабинет посетителей **не
   принимает**. Перевозка идёт: скопировать → проверить (verify) → и только теперь «открыться».
2. 🕐 **Штамп момента открытия (`MoveTs`).** В момент завершения переезда зоны ставим **отметку
   времени** — «эта зона официально здесь с момента T».
3. 🚫 **Старые пропуска не действуют.** Если посетитель пришёл с временем `ts < MoveTs` (по старой
   карте, до переезда) — его **вежливо разворачивают** («тут этих данных на тот момент ещё не было»).
   Он перечитает актуальную раскладку и придёт по новому адресу.
4. ✅ **Новые — проходят.** Чтения с `ts ≥ MoveTs` спокойно обслуживаются с нового диска.
5. 🗑️ **Старый кабинет освобождаем в конце.** Исходник удаляется только после verify+fence
   (двухфазно, через delete-set) — никаких «дыр» посередине.

> Чертёж — из Dgraph (predicate-move: `tablet.MoveTs`, чтения с `ts < MoveTs` отклоняются). Это
> **epoch-fencing**, родственно манифест-fencing. Сочетается с [walk-resilver.md](walk-resilver.md),
> [disk-balancer.md](disk-balancer.md) и [two-phase-delete.md](two-phase-delete.md).

## Проверка понимания

> При переезде зоны блоков целевой диск ставит штамп **MoveTs**; чтение по «старому времени»
> (`ts < MoveTs`) **отклоняется** → никто не прочитает устаревшую раскладку и не попадёт в «дыру».
> Перенос = copy → verify → fence(MoveTs) → удаление исходника.

## Зачем это нам

Фаза 3 (resilver/rebalance): корректность чтений **во время** миграции на 60-HDD узле — ребаланс не
создаёт окон рассинхронизации.
