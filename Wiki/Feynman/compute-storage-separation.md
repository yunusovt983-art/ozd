# Compute/storage separation (Часть 3) — «методом Фейнмана»

## Проблема (на будущее)

Сейчас у нас **один сервер-демон** на 60 HDD. Что, если читателей станет очень много и одного
сервера не хватит? Хочется добавить «голов», не копируя 480 ТБ данных на каждую.

## Аналогия — одна библиотека, много читальных залов

Вместо того чтобы у каждого читателя был **свой склад книг** (дорого, N копий данных), делаем:
- 📚 **Один общий склад** (shared storage — наш content-addressed blockstore).
- 🪑 **Много читальных залов без своего склада** (stateless gateway'и) — каждый просто **ходит на
  общий склад** за книгами и отдаёт читателям.

Добавить нагрузку = поставить ещё один читальный зал, **не дублируя склад**. Это и есть
**разделение вычислений (gateway) и хранения (склад)** — модель PolarDB.

**А сложность согласованности?** У СУБД она огромна (книги-«доски» переписывают). У нас —
**почти нулевая**: книги **неизменяемы** (см. [immutability.md](immutability.md)). Любой зал берёт
**неизменный блок по CID** — двух версий не бывает, кэш не протухает, мирить читателей с писателями
не нужно. **Наша неизменяемость делает этот путь радикально проще, чем у любой СУБД.**

## Проверка понимания

> **Compute/storage separation** = один общий склад данных + много «читальных залов» (stateless
> gateway'ов) без собственной копии. Масштабируем чтение, добавляя залы, не дублируя 480 ТБ. А
> immutability блоков делает согласованность почти бесплатной.

## Зачем это нам

Это направление **Части 3** (горизонтальное масштабирование). В Части 1 — один демон; но дизайн
держим так, чтобы переход был возможен (pluggable backend, см. [polarvfs.md](polarvfs.md)).
