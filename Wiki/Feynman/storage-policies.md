# Storage policies: тиры + move-factor — «методом Фейнмана»

## Проблема

У нас 60 дисков разной «температуры»: быстрые NVMe (горячее), SSD (тёплое), большие HDD (холодное),
плюс `cold_path`/S3 (архив). Данные со временем **остывают**: свежие читают часто, старые — почти
никогда. Как **автоматически** перемещать сегменты между тирами — без ручных решений и без россыпи
отдельных правил (температура, TTL, балансировка дисков по отдельности)?

## Аналогия — склад с зонами: «у входа», «средние полки», «дальний ангар»

Кладовщик не решает про каждую коробку вручную — он живёт по **правилам зонирования**:

1. 🏷️ **Тиры (volumes) по порядку.** Объявляем упорядоченные зоны: `[hot(NVMe) → warm(SSD) →
   cold(HDD) → archive(S3)]`. Новое кладём в первую зону, где влезает.
2. 📊 **move-factor: зона заполнилась → переливаем дальше.** Если в зоне свободно меньше порога
   (напр. <10%) — фон **переносит сегменты в следующую зону**, пока не освободит. Само, без оператора.
3. 📐 **Size-gate: крупное — сразу в дальний ангар.** Сегмент больше лимита тира не кладём в hot —
   отправляем в более ёмкий cold (мелкое/горячее не выселяется крупным).
4. 🗓️ **move-TTL: по возрасту вниз.** Правило «через 30 дней → warm, через 90 → cold» само спускает
   сегменты по каскаду.
5. 🚚 **Перенос безопасно: clone → swap → delete.** Скопировать на новый тир, атомарно переключить,
   старое удалить (как [disk-balancer](disk-balancer.md)).

> Чертёж из ClickHouse (StoragePolicy: volumes + `move_factor` + `max_data_part_size` per volume + TTL
> MOVE TO VOLUME). **Объединяет** в один конфиг температурный тиринг (RocksDB), [disk-balancer](disk-balancer.md)
> и [декларативные drop-rules](#) (Druid).

## Проверка понимания

> **Storage policies** = объявить **упорядоченные тиры** (NVMe→SSD→HDD→S3) и **автоматически**
> переливать сегменты вниз по каскаду — по **заполнению** (move-factor), **размеру** (size-gate) и
> **возрасту** (move-TTL). Единый декларативный каркас вместо ручного тиринга.

## Зачем это нам

Фаза 5: единый декларативный тиринг на 60 дисках — горячее на NVMe, холодное на HDD/S3, само и предсказуемо.
