# Запрос по диапазонам чанков (chunk-range) — «методом Фейнмана»

## Проблема

При resilver/докачке у нас уже есть **часть** данных, не хватает кусков в середине. Просить у
источника **весь** объект — расточительно (гоним то, что уже есть). Как попросить **ровно
недостающее**?

## Аналогия — список «чего не хватает» вместо «пришлите всё заново»

Собираете коллекцию марок: у вас есть № 1–5 и 8–10, нет 6–7.

1. 📋 **Просим только пропуски.** По [карте докачки](partial-bitfield.md) считаем `missing()` и шлём
   запрос «дайте чанки 6–7 объекта H» — не весь объект.
2. 🗜️ **Компактный формат.** Диапазоны кодируем **дельтами границ** (varint), напр. `[2,5,3,1]` =
   «пропусти 2, возьми 5, пропусти 3, возьми 1». Запрос крошечный, чанки — по 1024 байта (BLAKE3).
3. 🔁 **Resumable по природе.** Оборвалось снова — пересчитали `missing()`, попросили остаток. Нулевой
   повтор работы.

> Чертёж из iroh-blobs (`protocol/range_spec.rs`: `RangeSpec`/`ChunkRangesSeq`, `GetMany` для пачки).
> Связка с [bitfield](partial-bitfield.md), [verified-streaming](verified-streaming.md),
> [multi-source](multi-source-fetch.md).

## Проверка понимания

> **Chunk-range запрос** = просить у источника **только недостающие** чанки (`missing()` по bitfield),
> компактно дельта-кодируя диапазоны → нет передачи уже имеющегося, естественно resumable.

## Зачем это нам

Фаза 3/4: трафик resilver/Bitswap ≈ объёму недостающего, а не блока/набора целиком.
