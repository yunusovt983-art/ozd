# Ops-планировщик (scylla-manager) — «методом Фейнмана»

## Проблема

Кроме обслуживания клиентов, демону надо регулярно делать **хозяйственные дела**: проверять
целостность (scrub), доводить число копий (resilver-проверка), делать бэкап в холодное/облако. Если
запускать их **прямо в горячем коде** вперемешку — они мешают, и легко забыть/продублировать.

## Аналогия — отдельный «завхоз» с расписанием

В большом доме есть **завхоз** с **календарём**: «по вторникам — проверка пожарных датчиков, раз в
месяц — инвентаризация, по ночам — резервное копирование». Жильцы (клиенты) этим не заняты — у них
свои дела; завхоз тихо делает фоновую работу **по расписанию**, не дёргая жильцов.

Так и **ops-планировщик** (у ScyllaDB это отдельный сервис **scylla-manager**): **вынесенный
cron-подобный планировщик** фоновых задач — `scrub`, периодическая проверка реплик, `backup` в
S3/cold_path, `restore`. Он **отделён от горячего пути** и работает под общим бюджетом IO
([Forseti](forseti.md)), чтобы не мешать клиентам.

## Проверка понимания

> **Ops-планировщик** = отдельный «завхоз с расписанием»: по графику запускает фоновые
> хозяйственные задачи (scrub / проверка копий / бэкап), вынесенные из горячего пути и
> ограниченные по IO, чтобы не мешать клиентским запросам.

## Зачем это нам

Фаза 5: единое место для регулярного обслуживания пула (scrub/resilver/backup в cold_path) — вместо
разбросанных по коду таймеров. Прообраз — scylla-manager.
