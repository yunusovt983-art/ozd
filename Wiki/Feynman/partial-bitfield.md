# Карта докачки (bitfield + spillover) — «методом Фейнмана»

## Проблема

Восстановление реплики (resilver) или удалённый fetch **оборвался на середине**. Как при
возобновлении не качать **сначала**, а догрузить только недостающее? И как при этом не держать кучу
незавершённых передач **в RAM**?

## Аналогия — чек-лист сборки и стол, который не безразмерный

1. ☑️ **Чек-лист полученных кусков (bitfield).** Для объекта ведём карту «какие чанки уже есть».
   Оборвалось — при возобновлении смотрим в карту и просим **только пропуски** (`missing()`).
2. 📐 **Известные размеры (sizes).** Рядом — «правый известный размер», чтобы понимать общий объём
   даже при кусках, пришедших не по порядку.
3. 🧱 **Карта — производная.** Если карта потерялась — **восстановим из данных** (проверкой
   присутствующих чанков), как и индекс восстановим обходом. Не священная.
4. 🪑 **Стол не безразмерный (spillover).** Пока кусков мало — держим в RAM; перевалило за порог —
   **сбрасываем sparse на диск** (только записанные куски + карта). Массовый rebuild не съедает память.

> Чертёж из iroh-blobs (`partial_mem_storage.rs`, `bao_file.rs`: PartialMem → PartialFile, bitfield
> реконструируется из outboard). Связка с [chunk-range](chunk-range-fetch.md) и [verified-streaming](verified-streaming.md).

## Проверка понимания

> **bitfield + spillover** = карта «что уже докачано» (просить только пропуски, восстановима из
> данных) + перенос незавершённой передачи из RAM на диск по порогу → resumable resilver без жора памяти.

## Зачем это нам

Фаза 3: возобновляемый resilver/fetch с места обрыва; ограничение RAM при массовом восстановлении.
