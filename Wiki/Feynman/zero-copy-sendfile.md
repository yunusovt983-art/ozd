# Zero-copy sendfile — «методом Фейнмана»

## Проблема

Клиент по сети просит блок. Обычная отдача = «прочитать файл в память приложения → записать из памяти
в сокет». Это **двойное копирование** через user-space (диск→page-cache→буфер приложения→буфер
сокета) + лишний CPU. На потоке в гигабайты/с по 60 HDD это заметная трата. Как отдать байты **минуя
приложение**?

## Аналогия — трубопровод напрямую, а не «ведром через комнату»

Чтобы перелить воду из бочки в цистерну, глупо носить **вёдрами через комнату** (диск → приложение →
сокет). Проложи **трубу напрямую**:

1. 🚰 **sendfile = труба ядра.** Просим ОС: «перекинь диапазон `[offset, len]` файла **прямо в
   сокет**». Данные идут **из page-cache в сетевую карту внутри ядра** (DMA), **не заходя** в память
   приложения. Одна команда (`transferTo`) вместо read+write.
2. 🎯 **Только для непрозрачного, уже проверенного.** Подходит, когда отдаём **сырой байт-диапазон**
   блока, который **уже верифицирован** (CRC сошёлся при scrub/записи). Если нужна **потоковая
   проверка по ходу** — это другой путь ([verified-streaming](verified-streaming.md)).
3. ⚡ **Меньше CPU и копий → выше throughput.** На 60 HDD освобождённый CPU = больше параллельных
   отдач; нет давления на heap.

> Чертёж из Kafka (`FileRecords.transferTo` → `FileChannel.transferTo` = syscall `sendfile`/`splice`).
> Дополняет [serve-off-disk](#) (там — verified-поток с flow-control); sendfile — для чистого
> непрозрачного диапазона.

## Проверка понимания

> **Zero-copy sendfile** = отдавать сырой диапазон сегмента **из page-cache прямо в сокет** командой
> ядра (`transferTo`), минуя память приложения → меньше копий/CPU. Для уже-проверенных непрозрачных тел.

## Зачем это нам

Фаза 4: высокий throughput отдачи блоков по сети (Bitswap/gateway) при минимуме CPU на 60 HDD.
