# Параллельный бэкап (DFS: файл на диск + summary) — «методом Фейнмана»

## Проблема

Бэкап 60 дисков в `cold_path`/S3. Делать **один большой файл последовательно** — медленно (не
используем параллелизм 60 шпинделей) и хрупко (оборвалось — непонятно, что цело). Как бэкапить
быстро и с понятной целостностью?

## Аналогия — каждый отдел сдаёт свою папку, секретарь сшивает оглавление

1. 🗂️ **Файл на диск, все параллельно.** Каждый из 60 дисков (shared-nothing) пишет **свой файл
   бэкапа независимо** — полная загрузка всех шпинделей разом.
2. 📑 **Summary последним.** Когда все дольки готовы — пишем **файл-оглавление (summary)**, который
   фиксирует набор файлов и топологию. Он идёт **последним** → его наличие = «бэкап целостен».
3. ☁️ **Backend на выбор.** Назначение pluggable: локальный `cold_path` / S3 / GCS / Azure.
4. 🔒 **Под консистентным обходом.** Снимок идёт **поверх MVCC** индекса (redb) — конкурентная запись
   не рвёт обход, fork и удвоение RAM не нужны (см. [иммутабельность](immutability.md)).

> Чертёж из Dragonfly (`save_stages_controller.rs`: DFS — файл на шард + summary, cloud-backends).

## Проверка понимания

> **DFS-бэкап** = каждый диск пишет свой файл **параллельно**, затем **summary-оглавление последним**
> (его наличие = целостность); под MVCC-обходом, без fork; backend cold_path/S3/GCS/Azure.
