# Pre-zeroed фикс-слоты в журнале (zero = конец) — «методом Фейнмана»

## Проблема

Наш индекс-журнал (WAL) на NVMe пишется непрерывно. После **краха** надо быстро понять: **где
кончаются настоящие записи**, а где — мусор/недописанный хвост? И ещё хочется **мгновенно найти
N-ю запись** без чтения всех предыдущих.

## Аналогия — тетрадь в клеточку, заранее стёртая ластиком

Возьми тетрадь, где под записи отведены **строки одинаковой высоты**, и перед началом ты **стёр всё
ластиком** (страница чистая, нули).

1. 📏 **Одинаковые строки (фикс-слоты).** Каждая запись — ровно `N` байт в своём слоте
   (`term|index|смещение|тип`). Тогда «5-я запись» = просто отступ `5×N` — **нашёл за один шаг**,
   без чтения предыдущих (O(1), чистая арифметика).
2. 🧽 **Заранее всё занулено.** При создании файла регион слотов **забит нулями**. Это бесплатный трюк.
3. 🔚 **Ноль = «здесь конец».** После краха идём по слотам, **пока поле не ноль**. Первый занулённый
   слот = граница: дальше валидных записей нет. **Зануление само работает маркером конца** — не нужен
   отдельный «счётчик записей», который мог не успеть сохраниться.
4. 💾 **mmap переживает падение процесса сам.** Файл отображён в память; если упал **процесс** (не
   сервер) — ОС всё равно допишет страницы. `msync` (дорогой на диске) делаем **выборочно** — только
   ради защиты от полного выключения питания.

> Чертёж — из Dgraph raftwal (фикс-слоты 32Б в pre-zeroed 1МБ-регионе + данные после; mmap-survives-
> crash). Третий детектор хвоста рядом с [pack-segments.md](pack-segments.md) `flushOffset` и
> eof-маркером [group-commit.md](group-commit.md).

## Проверка понимания

> Журнал = **строки одинакового размера, заранее занулённые** → N-ю запись находим арифметикой (O(1)),
> а **первый нулевой слот = конец** валидных данных после краха. mmap спасает от падения процесса,
> msync — лишь от выключения питания.

## Зачем это нам

Фаза 1/6 (recovery index-tier): мгновенная адресация записей журнала + дешёвый детектор торн-тейла
после краха, без отдельных счётчиков.
