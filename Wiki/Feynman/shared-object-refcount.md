# Shared-object refcount: последний гасит свет — «методом Фейнмана»

## Проблема

В Части 3 несколько stateless gateway'ев работают над **общим** холодным хранилищем (cold_path/S3).
Один и тот же сегмент нужен нескольким из них. Если каждый **скопирует себе** — лишние копии и трата
места. Если хранить **один**, то кто и когда его удалит? Удалит один gateway — а другой ещё читает →
потеря.

## Аналогия — общая комната и счётчик вошедших: последний выключает свет

В коворкинге одной переговоркой пользуются несколько команд. Не строят по переговорке каждой —
**одна на всех**, со счётчиком «сколько команд ещё ею пользуются»:

1. 🔗 **Один объект, много ссылок.** Сегмент в cold-store лежит **в одном экземпляре**; каждый
   gateway, которому он нужен, **ссылается** на него (не копирует). Появилась ссылка → `refcount++`.
2. ➖ **Отпустил — счётчик вниз.** gateway больше не нуждается → `refcount--`. Объект **не трогаем**,
   пока счётчик > 0 (кто-то ещё держит).
3. 💡 **refcount == 0 → последний удаляет.** Когда отпустил **последний** — объект физически
   удаляется из cold-store. Без «надгробий» (tombstone) и блокировок-аренд (lease) — просто счётчик.

> Чертёж из ClickHouse (zero-copy replication: `ref_count` в метаданных объекта; `createHardLink` →
> `ref_count++`, unlink → `ref_count--`, при `==0` → удалить из S3). Расширяет наш
> [инкрементальный backup + refcount](incremental-backup.md) (#107, там — между бэкап-точками) на
> **живые** gateway. ⚠️ Это **Часть 3** (несколько gateway); в Части 1 (один демон) не нужно.

## Проверка понимания

> **Shared-object refcount** = несколько gateway делят **один** cold-сегмент по ссылкам со счётчиком;
> `refcount++` при ссылке, `--` при отпускании, **последний (refcount=0) удаляет** объект. Без
> tombstone/lease.
