# Домен без IO

## Аналогия

Представь почтовую систему. Есть **правила** («каждое письмо доставляется
в 2 копиях в разные ящики») — и есть **физика** (курьер, грузовик, дорога).

Правила не зависят от того, КАК доставляется: курьером, дроном, голубем.
Они описывают ЧТО должно быть true.

## Что это значит в ozd

`ozd-domain` — это **книга правил**. Она говорит:

- «Существует штука `BlockStore`, которая умеет put/get/delete» — но НЕ
  говорит, что это файл на диске.
- «Существует штука `PlacementPolicy`, которая выбирает R дисков для ключа»
  — но НЕ говорит, что это HRW-хэш.
- «Если запись не добралась до W дисков — это `QuorumNotReached`» — но НЕ
  знает, были ли это HDD или NVMe.

Физические действия (открыть файл, записать байты, отправить HTTP) — это
**адаптеры**. Они живут в других крейтах:
- `ozd-engine` открывает файлы и пишет сегменты
- `ozd-ipfs` слушает HTTP и проверяет SigV4
- `ozd-zfs` вызывает `zpool status`

## Зачем разделение

Если правила перемешаны с физикой — нельзя:

1. **Тестировать правила без дисков** — в памяти, за микросекунды
2. **Менять физику** (ZFS → XFS → remote) не трогая правила
3. **Понимать инварианты** системы, не продираясь через IO-код

Когда `ozd-domain` компилируется **без единой зависимости на файловую
систему** — это гарантия: правила чисты, физика подключаема.

## Физическая аналогия

Как формула F=ma не зависит от того, кто именно толкает объект. Формула —
домен. Рука, пружина, ракетный двигатель — адаптеры.

## В коде

```rust
// ozd-domain: ЧИСТОЕ правило (никакого std::fs, никакого tokio)
pub trait BlockStore: Send + Sync {
    fn put(&self, key: &BlockKey, data: &[u8]) -> DomainResult<()>;
    fn get(&self, key: &BlockKey) -> DomainResult<Vec<u8>>;
}

// ozd-engine: АДАПТЕР (реализует правило через физику)
impl ShardEngine for DiskEngine {
    fn put(&self, key: &BlockKey, data: &[u8]) -> DomainResult<()> {
        // здесь: файл, CRC, redb, fsync — физика
    }
}
```

Домен определяет *контракт*. Адаптер выполняет *работу*.
