# Erasure-набор + перестановка шардов — «методом Фейнмана»

## Проблема

В Части 1 мы храним **две полные копии** блока (R=2) — это 2× места. Дешевле — **erasure coding**:
бьём данные на K кусков + добавляем M «контрольных» (паритет); теряем любые M из K+M — восстановим.
Но как разложить эти K+M шардов по 60 дискам, чтобы **не было «горячего» диска** и всё было предсказуемо?

## Аналогия — раздать страницы книги по сейфам, плюс «запасные» страницы

Книгу режем на 12 страниц (K) и делаем 4 «волшебных» страницы-паритета (M). Любых 12 из 16 хватит,
чтобы восстановить всю книгу. Куда класть 16 страниц?

1. 🗄️ **Сейфы — в группы (erasure-наборы).** 60 сейфов делим на наборы по ~16. Книга целиком живёт
   **в одном наборе** (16 сейфов) — не размазана по всем 60.
2. 🎯 **Какой набор — решает хэш.** `sipHash(имя, число_наборов, id_пула)` → номер набора. Детерминированно
   (как наш HRW по CID), стабильно при добавлении пулов.
3. 🔀 **Перестановка на каждую книгу.** Внутри набора 16 страниц раскладываем **по своей перестановке**
   для каждой книги: у книги A страница-1 в сейфе-0, у книги B — в сейфе-7. Эта перестановка
   (`distribution-array`) хранится в метаданных книги. → **ни один сейф не «первый» всегда** = нет горячего.
4. ➗ **Сколько паритета.** По размеру набора: 8+ дисков → **EC:4** (12 данных + 4 паритет). Overhead
   1.33× (vs 2× у mirror), переживает 4 отказа в наборе.
5. 🔧 **Чтение и ремонт.** Достаточно **любых K** страниц → собрать книгу (`reconstruct` из паритета).
   Запись считается успешной при write-кворуме (≈ K, +1 если K==M).

> Чертёж — из RustFS/MinIO (`sets.rs` sipHash→набор, `set_disk/metadata.rs` distribution-array,
> `storageclass.rs` default-parity). Расширяет [ydb-erasure.md](ydb-erasure.md) (block-4-2). **Часть 2**.

## Проверка понимания

> Данные = K кусков + M паритета (любые K из K+M → восстановление). Диски группируем в **наборы**;
> книга→набор по `sipHash`; внутри набора шарды раскладываем **перестановкой на объект** (нет горячего
> диска). 16 дисков → EC:4 (12+4, overhead 1.33×, держит 4 отказа).

## Зачем это нам

**Часть 2**: перейти с R=2 mirror (2× место) на erasure (1.33–1.5× место) — конкретный, проверенный в
RustFS/MinIO чертёж раскладки шардов по нашим 60 HDD.
